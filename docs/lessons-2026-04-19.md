# cam-flow: Lessons Learned & TODO from Production Runs

Date: 2026-04-19
Based on: RV32 ECC Verification workflow + CoreMark Benchmark workflow

---

## Part 1: Problems Encountered

### P1. Agent concept wrong in DSL
`do: agent claude` treats agent as an anonymous tool call. Should be `do: agent <name>` referencing a predefined sub-agent with its own identity, prompt, skills, tools, and model.

**Impact:** Every node's prompt is written from scratch in workflow.yaml. No reuse, no quality control, no skill loading.

**Fix:** Agent definition files in `~/.claude/agents/<name>.md` with: name, description, model, tools, skills, system prompt. Workflow.yaml only says which agent + what task.

### P2. No preflight check
Engine runs expensive nodes (30-60 min simulation) without first verifying prerequisites are met. CoreMark simv ran 30 minutes before we discovered core didn't boot.

**Impact:** Hours wasted on operations that were never going to succeed.

**Fix:** Add `preflight` field to DSL. Engine runs preflight (cheap, seconds) before node execution (expensive, minutes). Fail fast.

### P3. Verify conditions too weak
Verify checks "step completed" not "goal achieved":
- `test -f simv` → simv exists but doesn't work
- `grep -q finish simv.log` → sim finished but core was stuck
- Should be: `test -s trace.mon.log` → core actually executed instructions

**Impact:** Nodes pass verification but haven't actually accomplished their purpose.

**Fix:** Planner prompt rule: "verify must check the OUTCOME not the OUTPUT. Did we achieve the goal, not just produce a file?"

### P4. Planner lacks domain-specific rules
Generic planning rules don't know hardware workflow needs:
- Analyze DUT interface before building TB
- Validate DUT alive before running benchmarks
- Check memory width before selecting hex files
- Deterministic operations (smake, VCS) should be cmd not agent

**Impact:** Plans miss critical steps. Agent wastes time discovering what planner should have planned.

**Fix:** Domain-specific rule sets for planner: hardware/RTL, software/testing, deployment, research.

### P5. No feedback loop from execution to planner
When execution reveals plan is wrong (missing analyze_tb step, wrong hex width), only human intervention can fix. Engine retries same broken plan.

**Impact:** Retry budget wasted on a fundamentally flawed plan instead of re-planning.

**Fix:** Replan mechanism (future) — when node fails N times at L3+ escalation, suggest replan to user.

### P6. Agent score came from gold trace not own simulation
Agent found historical data and used it as the answer instead of running the actual test. Technically "got a score" but not from current design.

**Impact:** False results. Design could have changed, score is meaningless.

**Fix:** Verify conditions must check result origin. `grep -q "own simv" REPORT.md` or require specific log file from this run.

### P7. Node granularity too coarse
run_simv tried to: dump wave + analyze signals + fix boot + recompile + verify + run CoreMark. Too much for one agent in one timeout.

**Impact:** Agent times out mid-work. State not preserved between steps.

**Fix:** Planner rule: "One node = one independently verifiable deliverable. If a node needs to do X then check then do Y, split into two nodes."

### P8. verify template vars not resolved
Step 1 of ECC workflow: verify had `{{state.sim_dir}}` but template substitution didn't run on verify cmd. Verify executed `test -n "{{state.sim_dir}}"` literally.

**Impact:** Verify always passes (non-empty string) or fails for wrong reason.

**Fix:** Engine must resolve_refs() on verify cmd before executing it. (May already be fixed in latest code — need to verify.)

### P9. camflow-fix agents not cleaned up
Engine spawned 6 camflow-fix agents during calculator demo, none were removed after completion.

**Impact:** Agent clutter, resource waste, confusion.

**Fix:** Engine cleanup implemented (zero leftover). Also pending: camc rm must kill tmux + delete socket to prevent heal re-adoption.

### P10. Planner picked wrong methodology for some nodes
setup_tree got "systematic-coverage" when it should be "simplify-first". Methodology descriptions in planner prompt were too vague.

**Impact:** Agent gets wrong guidance for approach.

**Fix:** Updated planner prompt with clear decision guide per methodology. Verified with v3 regeneration.

## Part 3: Architectural Decisions Confirmed

1. **Stateless execution model** — each agent node starts fresh, context via state.json. Confirmed working in production.
2. **Plan/Execute separation** — planner decides, engine executes, engine never changes plan. Confirmed correct boundary.
3. **Creator/Planner/Engine roles** — Creator prepares + interacts with user, Planner makes deterministic decisions, Engine executes. No "foreman agent" needed.
4. **Two skills** — camflow-manager (full lifecycle) + camflow-runner (CLI mode). Confirmed working.
5. **File signal for completion** — node-result.json appearance, not auto-exit or idle detection. Confirmed reliable.
6. **verify + preflight** — two-layer validation: preflight before (can I start?), verify after (did I succeed?). New insight from production.

## Part 5: CoreMark follow-up (2026-04-20)

Second-phase run shipped a definitive score and surfaced four new
planner rules. Full writeup in
`docs/postmortem-coremark-2026-04-20.md`; summary:

### Final numbers
- Raw (face-value) score: **0.8754 CoreMark/MHz** (from
  `CoreMark Iterations: 1` in the log)
- Corrected: **1.7509 CoreMark/MHz** (matches gold reference ~1.8
  within ~3%)
- Measured IPC: **0.507** (cross-validated two independent ways,
  agree within 1%)
- Workflow: 32 steps, 75% step-success, ~75 min total

### The wasted loop
`validate_score_sanity` (500 K trace-line threshold) failed 5× on
a run whose full-sim tracer was disabled for wall-clock. Three
loops of `run_simv → validate → fix_tracer` before a manual
`--from investigate_score` broke it. Cost: ~25 min of retry budget
on the wrong problem.

### New planner rules (added to docs/postmortem-coremark-2026-04-20.md §4)
1. Any node that parses / validates benchmark OUTPUT must be
   preceded by a node that reads the TB and reports which fields
   are hard-coded, plusarg, or computed.
2. Before any simulation with `timeout > 5 min`, a short-run
   validation must exercise the same score parser the full run
   will use.
3. Every numeric output-extraction node MUST declare an
   `expected_range`; results outside it route to investigate, NOT
   back to a retry of the producing step.
4. Never line-count a log file the production run isn't guaranteed
   to populate. Range-check the final metric instead.

### Confirmed architectural decisions (reinforced)
- preflight + verify two-layer gate (§P2, §P3 from Part 1) paid
  off again — compile_tb, run_simv, parse_score each used preflight
  to skip rework across retries
- `camflow evolve report` trace rollup produced the timeline used
  to write the postmortem in minutes, not hours
- Read-only investigate nodes (produced 253-line INVESTIGATION.md)
  are first-class, not escape hatches — planner should generate
  them proactively on any result-out-of-range transition
