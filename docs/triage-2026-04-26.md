## 2. The self-destruct hypothesis

### 2.1 What I found

`src/camflow/backend/cam/agent_runner.py:142–160` (pre-existing,
unchanged by these 9 commits) defines:

```python
def cleanup_all_camflow_agents():
    """Belt-and-suspenders: remove every camflow-* agent from camc."""
    for aid in _list_camflow_agent_ids():     # camc --json list, filter startswith("camflow-")
        _rm_agent(aid)                         # camc rm <aid> --kill  (tmux kill-session)
```

`Engine._cleanup_on_exit` (`engine.py:564`, **new code from `885650b`'s
neighborhood but the call to cleanup_all is pre-existing**) calls this
unconditionally in the `finally` block of `Engine.run`.

The previous camflow-dev instance ran in a tmux session named
**`camflow-dev`**. That name matches `startswith("camflow-")`. The
moment that instance ran any code path that exited an `Engine.run` —
e.g. manually verifying its own work with `python -m camflow run …`
or letting an integration test call into Engine without the
`--no-steward` short-circuit — the finally block walked the camc
registry and `tmux kill-session`'d its own session.

### 2.2 Why this only blew up now

Two combined factors:

- The new commits added a strong incentive to manually run `camflow
  run` in the workspace: validate Steward spawn + events end-to-end.
- The new default `EngineConfig.no_steward=False` means that when the
  previous instance did `python -m camflow run examples/foo.yaml`, it
  also tried to spawn a real Steward via `camc run`. That gives the
  previous instance two ways to get bitten:
  1. Engine clean-exit → `cleanup_all_camflow_agents()` → kill self.
  2. Engine spawned a Steward, the test ended, Steward survived as
     an orphan; later cleanup pass kills self anyway.

### 2.3 Severity

**Critical.** The bug pre-exists, but the new code path makes it
trivially reachable. Anyone running `camflow run` from inside a
`camflow-*` tmux session will reproduce.

### 2.4 Recommended fix (do NOT apply yet — for Huailu's review)

Three layers, any one of which alone closes the hole:

1. `cleanup_all_camflow_agents()` should restrict its sweep to agents
   that belong to **this engine's `flow_id` or `project_dir`**. The
   registry (`agents.json`, new in `acaa767`) now records which agents
   the engine spawned — use that, not a name prefix glob.
2. Or: drop the belt-and-suspenders sweep entirely. The "(6 dead
   agents observed)" comment that justifies it pre-dates the registry;
   the registry now does this job better.
3. Or (defensive only): make the sweep skip any agent whose id matches
   the current process's tmux session.

I'd push for #1 — the others paper over the underlying ownership
confusion.

### 2.5 What's NOT in the new code

I audited every line of new code for `tmux kill`, `camc rm`, `camc
stop` calls:

- `src/camflow/registry/` — no subprocess, no kill calls.
- `src/camflow/cli_entry/ctl.py` + `ctl_read.py` — no mutating verbs
  registered. Only file reads. Cannot kill anything.
- `src/camflow/steward/spawn.py` — calls `camc run` and `camc status`,
  never `camc rm` or `camc stop`.
- `src/camflow/steward/events.py` — calls `camc send` only.
- `src/camflow/backend/cam/orphan_handler.py` — the new
  `_update_registry_if_present` only flips registry status; it does
  not introduce any new kill path. It calls the **existing**
  `_cleanup_agent` for the existing `ACTION_TREAT_AS_CRASH` branch,
  same as before.
- `src/camflow/backend/cam/engine.py` — new `_ensure_steward`,
  `_emit_steward_*` helpers wrap calls to the steward modules above.
  No kill.

So: **the new code did not write new tmux/camc-kill primitives**. It
exposed an existing one. That matters for the verdict — these commits
don't need to revert; they need a small follow-up that defangs the
existing hazard.

## 3. The 13 orphan workers hypothesis

The brief says "13 orphan worker agents in pytest temp directories
(`/tmp/pytest-of-hren/pytest-XXX/`)". I have two candidate causes:

### 3.1 Hypothesis A (most likely): real `camc run` from default-Steward path

`EngineConfig.no_steward` defaults to `False`. The integration tests
that touch `Engine.run` (e.g. `tests/integration/test_cmd_only.py` and
the resume tests) construct an Engine without setting `no_steward`
either way. The new wiring in `885650b` then calls
`spawn_steward(project_dir, ...)` which calls
`_default_camc_runner(...)` which is **a real `subprocess.run([camc,
"run", ...])`**.

If pytest ran integration tests during the previous instance's session,
each test would have spawned a real Steward in its `tmp_path`. The
test would have ended; nothing teardown'd the Steward; it lingered.

This explains *Stewards* in tmpdirs, but the brief said "worker"
agents specifically. Either:
- The brief used "worker" loosely (Stewards are also camc agents), or
- Some test ran further than I think (resumed flow, ran a node,
  spawned a `camflow-build-…` worker), or
- The previous instance manually ran `camflow run` against an example
  workflow and the agents are real workers.

### 3.2 Hypothesis B: manual end-to-end test by previous instance

Given that the work crossed Phase 1 → Phase 3 and the previous
instance was clearly running things, it likely ran something like
`python -m camflow run examples/hello.yaml` from the workspace to
verify the new Steward path. That would spawn:
- 1 Steward (named `steward-<id>` — does NOT match `camflow-*`).
- N workers (named `camflow-<node>-<id>` — DOES match `camflow-*`).

Workers were swept by `cleanup_all_camflow_agents()` in the engine's
own `finally`, **including the parent camflow-dev session**. By the
time aicli looked, the workers from the example flow were *probably*
gone too — so the 13 orphans we found likely include both manual-run
leftovers AND test leftovers. I can't distinguish without access to
the camc list at the time.

### 3.3 What this means for triage

- The 9 commits do not contain a code path that intentionally creates
  orphan workers.
- The 9 commits **do** include a default that makes any unmocked
  `Engine.run()` invocation spawn a real Steward.
- Tests in the new code (`test_engine_steward_wiring.py`,
  `test_steward_spawn.py`, `test_steward_events.py`) are unit tests
  with injected `camc_runner` / `camc_send` — those are clean.
- The integration tests touched in the working tree
  (`test_cmd_only.py`, `test_stateless_loop.py`, `test_resume_clean.py`)
  are **only** filter-fix changes (adapt to the new `kind` field).
  They are unchanged in their subprocess behavior.
- **The smoking gun is therefore the default flag, not a test bug.**

Recommendation: flip `no_steward` default to `True` until Phase A
ships fully, OR gate Steward spawn on `CAMFLOW_ENABLE_STEWARD=1` env
var, OR add a pytest autouse fixture in `conftest.py` that forces
`no_steward=True` for every Engine constructed during tests. (See
§6.)

## 5. Design doc evaluation

Verdict: **sound, keep**. A few minor inconsistencies to clean up
before §16's "files added/changed" gets used as a Phase 2 punch list.

### 5.1 What's good

- The trust model (§3) is the right framing. Separating
  intelligence from execution is exactly the spine this should sit on.
- Steward as project-scoped (not flow-scoped) and the `flow_idle` /
  archive fold pattern matches the user's "I want to ask 'how did the
  last flow go?' days later" requirement.
- The autonomy presets + per-verb override + confirm queue is small
  and ergonomic. The default mapping (kill-worker autonomous,
  spawn/skip/replan confirm) is defensible.
- Phased delivery (A foundations / B agency / C smooth-mode) is the
  right cut, and Phase A is what's actually been written.

### 5.2 Inconsistencies to fix

1. **Steward boot prompt promises `kill-worker` and other mutating
   verbs that won't exist until Phase B.** §7.7 of the design even
   says this is the spawn-time prompt that's never regenerated. So
   first-Steward ever will have stale tool docs. Either:
   - Make the prompt regen on Phase-B rollout, OR
   - Phase A's prompt should list only the 5 read verbs and add a
     "more verbs landing soon" note.
2. **Two near-overlapping closed sets** for "what events exist":
   - `tracer.EVENT_KINDS` (16 entries — broad: file ops, locks,
     watchdog, plus events that are also Steward events).
   - `steward.events.EVENT_TYPES` (13 entries — narrow: lifecycle the
     Steward actually subscribes to).
   They will drift over Phase B. Pick one source of truth or
   document the relationship explicitly.
3. **`flow_id` semantics across resume** (see 4.4 above) — design §12
   says `flows_witnessed` correlates flows; but `_init_runtime_state`
   regenerates flow_id on every `camflow run`. Decide which kind of
   "flow" is being correlated.
4. **§16 lists files like `chat.py`, `steward.py` (CLI),
   `agent_planner.py`** that are not in any commit. That's expected
   (Phase A is partial) but the doc should mark them.
5. **`camflow chat` docs in §8.2** say
   `camflow chat --history` and `--inbox`; these don't exist yet.
   Fine for the design doc, but Phase 2 plan should call them out as
   pending.

### 5.3 Open questions still open

OQ-1 through OQ-11 in §15 mostly have recommendations marked.
Implementation in the 9 commits silently picked Huailu's
recommendations as if they were already approved (e.g. OQ-1 = opt-out:
`--no-steward` exists). That's fine in spirit but the spec should
explicitly say "Huailu approved as of <date>" before Phase 2 freezes
them.

## 6. Recommendations (for Huailu's review only — no action taken)

In rough priority order:

### 6.1 Before any push — fix the self-destruct hazard

Pick one of:

- **(preferred)** Replace `cleanup_all_camflow_agents()`'s name-prefix
  glob with a registry-driven sweep that only kills agents the
  current engine spawned (use `agents.json` filtered by
  `flow_id == self.state["flow_id"]`).
- Drop the belt-and-suspenders sweep entirely. The orphan handler +
  per-flow registry are stronger guarantees than name globbing, and
  the original "(6 dead agents observed)" rationale predates them.
- **(weakest, but cheap)** Add a guard at the top of
  `cleanup_all_camflow_agents` that skips any agent whose id matches
  this process's tmux session.

### 6.2 Before any push — close the orphan-Steward leak

Pick one of:

- Flip `EngineConfig.no_steward` default to `True` until Phase A's
  end-to-end integration test ships and demonstrates Steward cleanup
  works in tmpdirs.
- Gate Steward spawn on `CAMFLOW_ENABLE_STEWARD=1` env var.
- Add an autouse fixture in `tests/conftest.py` that forces
  `no_steward=True` for every Engine instance (and verify any test
  that *wants* a real Steward sets `monkeypatch.setenv(...)` first).

### 6.3 Before Phase B — prompt regen on verb rollout

The Steward boot prompt (`steward-prompt.txt`) is written once at
spawn and never regenerated. Phase B introduces 5+ new verbs. Either
(a) regenerate the prompt when verb set changes, or (b) make the
prompt a runtime-loaded "tools" section that references
`camflow ctl --help` rather than enumerating verbs.

### 6.4 Phase 2 plan should call out these changes

- Add `cleanup_all_camflow_agents()` rework to Phase A's punch list.
- Add an integration test that runs `camflow run --no-steward
  examples/hello.yaml` from a non-`camflow-*` tmux session AND from a
  `camflow-test-victim` tmux session, verifying the second one isn't
  killed (this is the exact regression we want to never hit again).
- Add a `tests/conftest.py` fixture that asserts no leftover
  `steward-*` or `camflow-*` agents survive a test session.
